# Feature Specification: Live Pattern Alerts & Paper Trade Execution

**Feature Branch**: `017-live-pattern-alerts`
**Created**: 2026-03-08
**Status**: Draft
**Input**: User description: "Pattern alerts — notifications when a pattern triggers on live data. Live paper trading integration — auto-execute paper trades when patterns trigger."

## User Scenarios & Testing

### User Story 1 - Pattern Scanner with Alerts (Priority: P1)

Jordan has several backtested patterns (e.g., "Pharma News Spike Dip") that he wants to monitor against live market data without keeping a terminal window open all day. He runs a scan command that checks all his active patterns against recent market data. When a pattern's trigger conditions are met (e.g., MRNA spikes 7% on high volume), Jordan receives a notification with the details so he can decide whether to act.

**Why this priority**: Without alerting, Jordan must manually run backtests or watch charts to spot opportunities. This is the foundational capability that makes Pattern Lab actionable in real time.

**Independent Test**: Set a pattern to `paper_trading` status, run the scanner against recent market data, and verify that triggers are detected and alerts are persisted with full context (ticker, trigger type, pattern name, price data).

**Acceptance Scenarios**:

1. **Given** a pattern in `paper_trading` status with quantitative trigger rules, **When** the scanner runs and a ticker meets the trigger conditions in recent market data, **Then** an alert record is created with the pattern name, ticker, trigger details, timestamp, and recommended action.
2. **Given** multiple patterns in `paper_trading` status, **When** the scanner runs, **Then** all patterns are evaluated against their respective tickers and each triggered alert is recorded independently.
3. **Given** a pattern in `paper_trading` status, **When** no tickers meet the trigger conditions, **Then** no alerts are created and the scan completes silently.
4. **Given** the scanner has already alerted on a specific trigger event, **When** the same trigger conditions persist on the next scan, **Then** no duplicate alert is created for the same event.

---

### User Story 2 - Alert Review and History (Priority: P2)

Jordan wants to review recent alerts to see what patterns have triggered, when they triggered, and what happened to the underlying stock since the alert. He can view a list of recent alerts from the CLI or through Claude Desktop (MCP), filter by pattern or ticker, and see whether he acted on each alert.

**Why this priority**: Alerts are only useful if Jordan can review them at his convenience. A persistent alert log lets him check in between meetings or at end of day without missing opportunities.

**Independent Test**: After the scanner has generated alerts, run the alert review command and verify alerts are displayed with full context, filterable by pattern or ticker, and show the current status of each alert.

**Acceptance Scenarios**:

1. **Given** alerts have been generated by the scanner, **When** Jordan runs the alert list command, **Then** he sees a chronological list of alerts with pattern name, ticker, trigger time, trigger details, and status (new, acknowledged, acted_on, dismissed).
2. **Given** alerts exist for multiple patterns, **When** Jordan filters by a specific pattern or ticker, **Then** only matching alerts are shown.
3. **Given** an alert exists, **When** Jordan acknowledges or dismisses it, **Then** the alert status is updated and it no longer appears as "new" in future listings.
4. **Given** alerts have been generated, **When** Jordan asks Claude Desktop about recent pattern triggers, **Then** Claude can retrieve and summarize the alerts via an MCP tool.

---

### User Story 3 - Auto-Execute Paper Trades on Trigger (Priority: P3)

Jordan wants certain high-confidence patterns to automatically submit paper trade orders when triggered, without waiting for manual review. When auto-execution is enabled for a pattern and a trigger fires, the system submits a paper trade order through the broker's paper trading environment, logs the trade, and notifies Jordan that a trade was executed on his behalf.

**Why this priority**: Builds on US1 and US2 to close the loop from detection to execution. Only viable after alerting is proven reliable. Requires explicit opt-in per pattern and respects all existing safety controls.

**Independent Test**: Enable auto-execution on a pattern, run the scanner when trigger conditions are met, and verify a paper trade order is submitted, the trade is recorded, and an alert is generated noting the auto-execution.

**Acceptance Scenarios**:

1. **Given** a pattern with auto-execution enabled and the kill switch is off, **When** a trigger fires, **Then** a paper trade order is submitted to the broker's paper trading environment and the trade is recorded with `auto_executed` status.
2. **Given** a pattern with auto-execution enabled and the kill switch is on, **When** a trigger fires, **Then** no trade is submitted and an alert is created noting the trigger was detected but execution was blocked by the kill switch.
3. **Given** a pattern with auto-execution disabled (default), **When** a trigger fires, **Then** only an alert is created — no trade order is submitted.
4. **Given** the daily trade limit has been reached, **When** a trigger fires on an auto-execution pattern, **When** no trade is submitted and an alert notes the trigger was detected but execution was blocked by the daily trade limit.

---

### Edge Cases

- What happens when the scanner runs outside of market hours? Alerts should still be generated based on the most recent available market data, with a note that the market is closed.
- What happens when a ticker in a pattern's watchlist has no recent price data? The scanner skips that ticker and logs a warning, without failing the entire scan.
- What happens when the same pattern triggers on multiple tickers simultaneously? Each ticker generates its own independent alert and (if auto-execute is enabled) its own independent trade.
- What happens when the broker API is unreachable during an auto-execution attempt? The alert is still created, but the trade submission is marked as failed with the error reason. No retry is attempted automatically.
- What happens when a pattern is retired while alerts exist? Existing alerts remain in history but no new scans evaluate that pattern.

## Requirements

### Functional Requirements

- **FR-001**: System MUST scan all patterns in `paper_trading` status against recent market data when the scan command is invoked.
- **FR-002**: System MUST persist each triggered alert as a record with: pattern ID, pattern name, ticker, trigger timestamp, trigger details (price change, volume, conditions met), and recommended action type.
- **FR-003**: System MUST prevent duplicate alerts for the same trigger event on the same ticker within a configurable cooldown period (default: 24 hours).
- **FR-004**: System MUST provide a CLI command to list, filter, acknowledge, and dismiss alerts.
- **FR-005**: System MUST provide an MCP tool to retrieve recent alerts for use in Claude Desktop conversations.
- **FR-006**: System MUST support an auto-execution flag per pattern that, when enabled, automatically submits paper trade orders upon trigger detection.
- **FR-007**: System MUST check the kill switch and daily trade limit before auto-executing any trade.
- **FR-008**: System MUST log all scanner runs, alert generation, and auto-execution events to the audit log.
- **FR-009**: System MUST allow the scanner to be run as a one-shot command (scan once and exit) or on a recurring schedule (scan every N minutes until interrupted).
- **FR-010**: Alerts MUST include enough context for Jordan to evaluate the opportunity without needing to run a separate backtest (trigger price, volume multiple, pattern win rate from most recent backtest).

### Key Entities

- **Alert**: A notification that a pattern's trigger conditions have been met on live market data. Has a lifecycle: new, acknowledged, acted_on, dismissed.
- **Scanner Run**: A single invocation of the pattern scanner, recording which patterns were evaluated, how many triggers fired, and the scan timestamp.
- **Auto-Execution Setting**: A per-pattern flag indicating whether triggered patterns should automatically submit paper trades. Defaults to off.

## Success Criteria

### Measurable Outcomes

- **SC-001**: Scanner evaluates all active patterns and generates alerts within 60 seconds of invocation for up to 20 patterns with 10 tickers each.
- **SC-002**: Zero duplicate alerts for the same trigger event within the cooldown period.
- **SC-003**: All auto-executed trades respect the kill switch and daily trade limits with 100% reliability.
- **SC-004**: Jordan can review the last 7 days of alerts in under 10 seconds via CLI or Claude Desktop.
- **SC-005**: Alert history is retained for at least 90 days for pattern performance tracking.

## Assumptions

- Scanning uses existing cached market data infrastructure (fetch_and_cache_bars) and the same trigger detection logic used in backtesting.
- Desktop notifications are out of scope for the MVP; alerts are persisted to the database and viewable via CLI and MCP tools. Webhook/email notifications can be added later.
- The scanner reuses the existing trigger detection functions rather than building new detection logic.
- Auto-execution uses the existing paper trading order submission flow.
- The recurring scan mode uses a simple polling loop (scan, sleep, repeat) rather than a separate daemon or job scheduler.
- Only patterns in `paper_trading` status are scanned; `draft`, `backtested`, and `retired` patterns are ignored.
