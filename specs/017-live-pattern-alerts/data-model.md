# Data Model: Live Pattern Alerts & Paper Trade Execution

**Feature**: 017-live-pattern-alerts
**Date**: 2026-03-08

## New Table: `pattern_alert`

Stores alerts generated by the pattern scanner when trigger conditions are met.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | INTEGER | PRIMARY KEY AUTOINCREMENT | Unique alert ID |
| pattern_id | INTEGER | NOT NULL, FK → trading_pattern(id) | Which pattern triggered |
| pattern_name | TEXT | NOT NULL | Snapshot of pattern name at alert time |
| ticker | TEXT | NOT NULL | Which ticker triggered |
| trigger_date | TEXT | NOT NULL | Date of the triggering market data (YYYY-MM-DD) |
| trigger_details_json | TEXT | NOT NULL | JSON with price_change_pct, volume_multiple, conditions met |
| recommended_action | TEXT | NOT NULL | Action type from pattern rule set (buy_call, sell_call, etc.) |
| pattern_win_rate | REAL | | Win rate from most recent backtest (for context) |
| status | TEXT | NOT NULL, DEFAULT 'new' | Alert lifecycle: new, acknowledged, acted_on, dismissed |
| auto_executed | INTEGER | NOT NULL, DEFAULT 0 | 1 if a paper trade was auto-submitted |
| auto_execute_result | TEXT | | JSON with trade_id, order_id, or error reason if blocked |
| created_at | TEXT | NOT NULL, DEFAULT datetime('now') | When the alert was created |
| updated_at | TEXT | NOT NULL, DEFAULT datetime('now') | When the alert status last changed |

### Indexes

- UNIQUE INDEX on `(pattern_id, ticker, trigger_date)` — prevents duplicate alerts for the same event
- INDEX on `status` — efficient filtering by alert state
- INDEX on `created_at` — efficient time-range queries

### Alert Status Lifecycle

```
new → acknowledged → acted_on
 └──→ dismissed
```

- **new**: Scanner just created the alert, Jordan hasn't seen it yet
- **acknowledged**: Jordan has seen the alert but hasn't decided whether to act
- **acted_on**: Jordan manually placed a trade based on this alert (or auto-execution succeeded)
- **dismissed**: Jordan reviewed and decided not to act (false positive, too late, etc.)

## Modified Table: `trading_pattern`

Add column to support per-pattern auto-execution.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| auto_execute | INTEGER | NOT NULL, DEFAULT 0 | 1 to enable auto paper trade on trigger |

## In-Memory Models

### AlertRecord (used in scanner output)

```text
pattern_id: int
pattern_name: str
ticker: str
trigger_date: str (YYYY-MM-DD)
trigger_details: dict
  - price_change_pct: float
  - volume_multiple: float
  - conditions_met: list[str]
  - latest_price: float
  - previous_close: float
recommended_action: str
pattern_win_rate: float | None
status: str (new | acknowledged | acted_on | dismissed)
auto_executed: bool
auto_execute_result: dict | None
  - trade_id: int (if successful)
  - order_id: str (if submitted)
  - blocked_reason: str (if safety check failed)
  - error: str (if API error)
created_at: str
```

### ScanResult (returned by scanner)

```text
scan_timestamp: str
patterns_evaluated: int
tickers_scanned: int
alerts_generated: int
alerts: list[AlertRecord]
auto_executions: int
auto_executions_blocked: int
```

## Relationships

- `pattern_alert.pattern_id` → `trading_pattern.id` (many alerts per pattern)
- `pattern_alert` is independent of `paper_trade` — an alert may lead to a paper trade (via auto-execute or manual action), but the relationship is tracked via `auto_execute_result.trade_id`, not a foreign key
- `pattern_alert` is independent of `backtest_result` — the `pattern_win_rate` is a snapshot, not a join
